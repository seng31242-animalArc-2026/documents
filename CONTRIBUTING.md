# Contributing

Thank you for contributing to this repository. This document is a placeholder
for the team's collaboration workflow and should be updated as project practices
are finalized.

## Branching Strategy

Use short-lived feature branches for all changes.

- `main` should contain stable, reviewed work.
- Create branches from `main`.
- Use descriptive branch names, such as `feature/add-project-outline`,
  `fix/update-document-format`, or `docs/add-readme`.
- Open a pull request before merging changes into `main`.
- Keep pull requests focused and small enough to review comfortably.

## Commit Convention

Use clear, concise commit messages that describe the intent of the change.

Recommended format:

```text
type: short description
```

Common commit types:

- `docs` - Documentation-only changes.
- `feature` - New functionality or content.
- `fix` - Corrections to existing work.
- `chore` - Maintenance tasks.
- `refactor` - Restructuring without changing behavior.

Examples:

```text
docs: add initial repository readme
chore: add baseline gitignore
fix: correct document formatting
```

## Review Expectations

- Confirm the change is within the scope of the pull request.
- Check for clear naming, readable formatting, and accurate documentation.
- Leave comments when something is unclear or needs follow-up.

Update this document when the team agrees on a more specific workflow.
